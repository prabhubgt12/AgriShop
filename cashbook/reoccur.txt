how to handle reocurring transactions

Handling recurring transactions properly is important for both personal and shopkeeper use. The goal is to make them automatic, reliable, and simple.

Below is a practical, developer-friendly approach.

1) What is a Recurring Transaction?

A transaction that repeats automatically at fixed intervals.

Examples:

Monthly rent

Salary

Subscriptions

Shop rent

Supplier payment

2) Data Model (Database Structure)

Create a separate table for recurring entries.

Table: recurring_transactions
Field	Type	Description
id	int	primary key
title	text	Rent, Salary, etc.
amount	double	transaction amount
type	text	credit or debit
category	text	optional
start_date	date	first occurrence
end_date	date	optional
repeat_type	text	daily, weekly, monthly, yearly
repeat_interval	int	every 1 month, 2 weeks, etc.
last_generated_date	date	last auto entry date
is_active	boolean	enabled/disabled
3) When to Generate Recurring Entries

There are two common methods:

Method A: On App Open (Recommended)

Every time the app opens:

Check all active recurring transactions.

See if any entries are due.

Auto-create missing transactions.

Example logic:

if today > last_generated_date:
    generate new entry
    update last_generated_date

Why this is best:

Works offline

No background service needed

Simple and reliable

Method B: Background Job

Use:

Android WorkManager

Daily scheduled task

Not recommended for simple apps, because:

Background restrictions

Battery optimizations

Can fail on some devices

4) Generation Logic (Monthly Example)
Example:

Amount: ₹5,000

Start date: 1 Jan

Monthly repeat

Last generated: 1 Feb

Today: 5 Mar

Steps:

Calculate next due dates:

1 Mar

Since 1 Mar < today:

Create transaction for 1 Mar

Update last_generated_date = 1 Mar

5) Prevent Duplicate Entries

Always check before creating.

Rule:
Before inserting, check:

Does transaction already exist for this date?

Use:

recurring_id + date as unique key

6) UI for Recurring Transaction Setup

In the “Add Entry” screen, add:

Toggle:
[ ] Repeat this transaction

If enabled, show:

Repeat:

Daily

Weekly

Monthly

Yearly

End date (optional)

7) User Control Options

Allow users to:

Pause recurring entry

Edit amount

Delete future occurrences

8) Example Flow

User creates:

Rent
₹10,000
Debit
Repeat: Monthly
Start: 1 Feb

App behavior:

1 Feb → auto entry

1 Mar → auto entry

1 Apr → auto entry

9) Recommended Minimal Version (MVP)

For first release, support only:

Monthly recurring

No end date

Generate on app open

This covers 80% of use cases.

10) Pseudocode (Simple Logic)
for each recurring in recurring_table:
    if recurring.is_active:
        next_date = calculate_next_date(recurring)

        while next_date <= today:
            if not exists_transaction(recurring.id, next_date):
                create_transaction(recurring, next_date)

            recurring.last_generated_date = next_date
            next_date = calculate_next_date(recurring)

Which transaction should be edited?

Answer:
User should not edit any of the past transactions.

Correct Concept

There are two different things:

1) Normal transactions (Jan, Feb, Mar)

These are actual records.
They must never be changed automatically.

2) Recurring rule

This is the template that creates future entries.

This is what the user must pause.
